# AUDIT-3 — API Compatibility and Idempotency Policy

**Marker:** `[[AI-CLI|AUDIT-3|IN_PROGRESS|REPO_CI_READY]]`

## Scope

This policy governs future API contract changes and retry behavior for SIDPRO's `/api/v1` HTTP API. It records a source-level decision; it does not claim that every historical response is already fully normalized or that a deployed client has accepted these terms.

## Compatibility Policy

1. `/api/v1` is the current public API prefix. Additive compatible changes may remain in `v1`.
2. Breaking changes require a documented migration decision before merge: introduce a new versioned route, retain a compatibility field/route for an agreed period, or obtain an explicit client cutover decision.
3. A breaking change includes removing or renaming fields/routes, changing field meaning/type, tightening a previously accepted request in a way that breaks valid clients, or changing authorization requirements.
4. New optional response fields are compatible only when they do not reveal tenant-sensitive data.
5. Input validation failures use the standard `VALIDATION_ERROR` envelope generated by `parseWithZod` or the global validation pipe. New domain validators must preserve this envelope unless a documented compatibility exception exists.
6. Auth, authorization, not-found, conflict, and domain-transition errors must use Nest HTTP semantics and must not expose secrets, raw database errors, tokens, or cross-tenant existence details.

## Idempotency and Retry Matrix

| Operation class | Current policy | Required implementation / evidence |
| --- | --- | --- |
| Read-only `GET` endpoints | Safe and idempotent by HTTP semantics; pagination queries are bounded globally when supplied. | Controller access and query validation tests. |
| Login, refresh, and 2FA | No generic idempotency key. Refresh replay/rotation behavior is domain-specific and must remain covered by auth regression tests. | Existing refresh-token/replay tests; AUDIT-4 review for abuse controls. |
| Tenant provisioning | No generic idempotency key yet. Tenant code uniqueness and transaction boundaries prevent duplicate committed tenant codes, but duplicate client retries require caller-visible handling before external exposure. | Authorization exception test plus future provisioning retry decision. |
| Finance ledger realization/reversal | No generic middleware key. Ledger/reference semantics are domain-specific; a generic retry header could create ambiguous accounting behavior. | Finance ledger invariants and AUDIT-5 database controls. Add a domain idempotency key only after finance owner defines reference uniqueness and retry result behavior. |
| Letter generation/PDF | No generic key. The workflow uses domain state and storage compensation; repeated generation must be assessed against the approved letter state and output retention rules. | Letter transition/PDF compensation tests; staging concurrency validation later. |
| Administrative create/update/delete | Do not promise automatic replay safety. Clients must not retry destructive mutations blindly until an operation-specific contract is documented. | Future client/API contract test when exposing automated integrations. |

## Design Rules for Future Idempotency Keys

A domain may introduce an `Idempotency-Key` only when the PR defines:

1. which route(s) accept it;
2. tenant and actor binding;
3. request canonicalization/hash behavior;
4. success and failure replay response behavior;
5. persistence, retention, and conflict behavior;
6. concurrency handling and audit logging;
7. regression tests for first request, exact retry, changed payload, cross-tenant reuse, and concurrent submission.

A generic middleware must not be introduced before those decisions exist because financial, provisioning, and document workflows do not share identical business semantics.

## Required API Change Record

A PR changing a versioned route, request schema, response schema, authorization contract, or idempotency behavior must include:

- affected route and consumer;
- compatibility classification: additive, compatible behavior change, or breaking;
- migration/cutover plan when breaking;
- validation and error-envelope impact;
- tenant/privacy impact;
- required regression, integration, and release evidence;
- roadmap/AUDIT-3 impact.

## Remaining Validation

Persistent staging must exercise retry/concurrency behavior for finance, tenant provisioning, letters, and public tracking under realistic clients/proxies before AUDIT-3 can close.
